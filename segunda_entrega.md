Cambios y avances del proyecto (Segunda entrega)
- Evolución de requisitos

En esta segunda entrega se mejoró la definición de los requisitos del sistema. Ahora se cuenta con una organización más clara entre requisitos funcionales y no funcionales, lo que permitió entender mejor qué debe hacer el sistema y cómo debe comportarse. También se logró mayor claridad en las necesidades del usuario, especialmente en el proceso de registro y consulta de asistencia. Los requisitos fueron ajustados para alinearse mejor con el objetivo principal del proyecto, que es facilitar el control de asistencia de manera eficiente.

- Refinamiento de artefactos
Durante esta fase se trabajó en mejorar los artefactos del proyecto, principalmente las historias de usuario. Se agregaron detalles como criterios de aceptación y se consideraron posibles excepciones, por ejemplo, cuando un estudiante no está registrado o se introduce un ID incorrecto. Además, se estructuraron mejor los requerimientos para que sean más claros. Estos cambios representan una mejora importante respecto a la entrega anterior.

- Bosquejos de interfaces
Se desarrollaron prototipos básicos de la interfaz del sistema, en este caso a través de un menú en consola. Aunque es una interfaz sencilla, cumple con el objetivo de permitir al usuario interactuar con el sistema de forma clara. Se siguieron principios de diseño simples como mostrar opciones claras y mensajes comprensibles para el usuario.

- Correspondencia con los requisitos
Los prototipos y el sistema desarrollado corresponden directamente con los requisitos definidos. Por ejemplo, el menú permite registrar estudiantes, marcar asistencia y consultar información, lo cual responde a las historias de usuario planteadas. Esto demuestra que el diseño sí está alineado con lo que se definió en la etapa de requisitos.

Validación de interfaces
La validación se realizó mediante pruebas del funcionamiento del sistema, simulando el uso real por parte del usuario. Durante estas pruebas se identificaron mejoras en la claridad de los mensajes y en el flujo del menú. A partir de esto se hicieron ajustes para que el sistema fuera más fácil de usar.

- Descripción del proceso
El desarrollo del proyecto se llevó a cabo por medio de etapas organizadas, similares a pequeños ciclos de trabajo. En cada etapa se realizaron actividades como análisis, diseño, programación y pruebas. Se cuenta con evidencia del avance en el repositorio, donde se pueden observar los cambios realizados en cada fase.

- Gestión del proceso
Para la organización del trabajo se definieron tareas específicas, como el desarrollo del código, pruebas y documentación. También se mantuvo un seguimiento del avance mediante el repositorio, donde se reflejan los cambios realizados. Esto permitió llevar un mejor control del proyecto y cumplir con los tiempos establecidos.

- Métrica de contribución individual
La contribución se dividió entre los integrantes del equipo asignando responsabilidades específicas. Cada integrante participó en el desarrollo del código, documentación y pruebas. La medición de la participación se puede observar a través de los cambios realizados en el repositorio.

- Organización y documentación del repositorio
Se creó un repositorio en GitHub donde se almacenan los avances del proyecto. Este repositorio está organizado por archivos y cuenta con documentación básica en formato Markdown. Además, se trabajó con actualizaciones constantes que reflejan el progreso del sistema.
